Platform modernization

From Windows Servers to OpenShift

How we are changing application hosting, delivery, operations, and ownership

Welcome everyone. This talk is intentionally broad: it explains why we are moving, what is changing technically, and how teams work together from development through production.

What is changing?

Then: n-tier Windows hosting

  • Long-lived virtual or physical servers
  • Applications depend on server-specific installs
  • Manual patching, configuration, and release steps
  • Large applications deployed as a unit

Now: container platform

  • OpenShift runs containerized workloads
  • Dependencies travel with the application image
  • Deployment is automated through pipelines and GitOps
  • Services evolve and scale independently
The key shift is not just a new hosting location. We are moving from server-centered thinking to application-centered and platform-centered thinking.

Why Windows servers became difficult

Aging

Operating systems, frameworks, certificates, and middleware drift out of support unless continually maintained.

Snowflakes

Server-specific registry changes, installed runtimes, COM components, drivers, or local files make environments hard to recreate.

Risk

Manual releases and hidden dependencies increase outage risk, audit effort, and recovery time.

Windows servers are not the problem by themselves. The issue is the operating model around long-lived servers: they collect state, exceptions, and undocumented dependencies over time.

The dependency problem

Old pattern Impact Container pattern
Install runtime on each server Version drift between environments Runtime included in image
Manual middleware setup Hard to rebuild reliably Configuration declared as code
Local files and machine state Deployments depend on history Images are immutable and repeatable
Shared server for many apps One app change can affect another Isolated containers and namespaces
Containers do not eliminate all dependencies. They make dependencies explicit, versioned, reviewed, and reproducible.

The platform schedules containers, manages rollout health, restarts failed workloads, and provides standard ingress, logging, metrics, and policy controls.

OpenShift becomes the standardized runtime layer. Teams package applications as images, and the platform runs those images consistently across environments.

Moving toward microservices

From monolith pressure

  • One release can require full regression
  • Scaling usually means scaling the entire app
  • Ownership boundaries are blurred

To service autonomy

  • Services own focused business capabilities
  • Teams deploy smaller changes more often
  • Services scale based on their own load profile

Microservices are a design choice, not a mandate. The first goal is container readiness; service decomposition should follow business value and operational readiness.

It is important not to equate OpenShift with instant microservices. The migration creates the foundation. Decomposition should be deliberate.

GitLab is the collaboration and code review point. Jenkins performs repeatable build and validation steps. Argo CD keeps the cluster aligned with the desired state stored in Git.

Promotion through environments

Lower environments

DEV, SIT, and UAT provide fast feedback for integration, testing, and business validation.

Controlled gates

Preprod and prod promotions use merge requests in a kustomization repository to review desired-state changes.

Production safety

Approvals, traceability, rollback plans, and observability protect critical services and customers.

Promotion is not a manual copy from one server to another. It is a reviewed change to configuration that says which application version should run in each environment.

Team responsibilities

Team Primary responsibility Examples
Application teams Build, test, and own app behavior Code, Dockerfile, service manifests, health checks
Ops / platform team Operate OpenShift safely and reliably Cluster capacity, upgrades, networking, platform policy
Ops / release approvers Control protected environment changes Secrets, preprod/prod merge requests, deployment windows
Security and governance Set guardrails and verify compliance Image scanning, access controls, audit evidence
The operating model is shared ownership. Application teams do not become cluster administrators, and Ops does not become the application development team. The handoffs are expressed through code, approvals, and platform services.

Secrets and configuration

What we avoid

  • Passwords in source code
  • Environment-specific rebuilds
  • Manual edits on servers

What we use

  • OpenShift secrets and controlled access
  • Environment overlays in kustomization
  • Audited change requests for protected environments
Configuration should vary by environment without rebuilding the application image. Sensitive values are managed through approved secret processes rather than committed into code repositories.

What success looks like

Faster

Smaller, automated releases reduce lead time from commit to environment.

Safer

Repeatable builds, reviews, health checks, and rollbacks lower deployment risk.

Cleaner

Applications carry their dependencies, reducing server drift and hidden coupling.

Success is measured by outcomes: less time spent nursing servers, fewer environment surprises, clearer ownership, and faster delivery of business value.

The hardest work is often discovery: identifying hidden dependencies, support constraints, data flows, certificates, scheduled jobs, and operational procedures.

A practical migration should start with assessment and prioritization. Not every application needs the same migration path. Some can be rehosted in containers; others need refactoring.

Key takeaways

  • We are moving from long-lived servers to repeatable container images.
  • Dependencies become explicit, versioned, and portable.
  • CI/CD and GitOps make releases reviewable, automated, and traceable.
  • Ops, application teams, security, and release approvers share a clearer operating model.
  • OpenShift is the foundation for safer modernization, including selective microservice adoption.
Close by reinforcing that this is a modernization of delivery and operations, not simply a data center relocation.
